# VILM Canonical Spec

## Name

VILM = Vertically Integrated Language Model

## Why VILM Instead Of SLM

The corridor engine is not just a smaller generic language model.
It is becoming a domain-fused intelligence stack.

That means the important feature is not only model size.
The important feature is vertical integration across:

- corridor logic
- proof gates
- typed state
- spatial interface
- investor/operator translation
- deployment surface

## Core Definition

A VILM is a language-mediated intelligence layer that is structurally fused to a specific operational stack.

It does not only "know about" a domain.
It is wired into the domain's:

- canonical concepts
- decision rules
- state objects
- interfaces
- workflows
- outputs

## Corridor VILM Stack

### 1. Logic Core

`CCL` is the engine substrate.

It holds:

- over/under corridor logic
- scenario A/B/C reasoning
- proof gates
- metabolic and infrastructure constraints
- investor and operator framing primitives

### 2. Spatial Application

`Corridor Studio` is the spatial operating surface.

It translates logic into:

- map views
- route and segment focus
- scenario switching
- presentation mode
- operator and investor interpretation

### 3. Product Shell

`Next.js / Vercel` is the deployment shell.

It turns the corridor intelligence into:

- shareable routes
- stable preview URLs
- app structure
- API endpoints
- investor-safe surfaces

### 4. Language Layer

The language layer is where the VILM lives.

It handles:

- corridor reasoning
- translation between technical and investor language
- structured summaries
- planning
- proof narration
- operator guidance

## Design Principle

The VILM must preserve corridor-specific intelligence.

It must not collapse into:

- generic dashboard language
- generic urban planning boilerplate
- polished product shell with no engine substance

## Milestone Ladder

### Milestone 1

Behavioral parity shell.

- preserve current Corridor Studio behavior
- preserve scenario logic
- preserve presentation mode
- preserve investor-safe sequencing

### Milestone 2

Typed state bridge.

- define CCL handshake objects
- move selected HUD text into structured state
- keep thesis-specific framing intact

### Milestone 3

Engine-backed application behavior.

- replace manual text with engine-fed outputs
- connect proof gates to real API objects
- support scenario comparison through structured state

### Milestone 4

True corridor VILM behavior.

- domain-grounded planning
- corridor-specific translation
- operator and investor mode adaptation
- learned and explicit logic working together

## Practical Test

If the system gets prettier but loses corridor-specific reasoning, it is not becoming a VILM.

If the system becomes more structured, more deployable, and more faithful to the corridor thesis at the same time, it is moving toward a VILM.
